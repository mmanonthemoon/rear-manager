**Models layer:**
Models should have minimal or no exception handling — they are simple DB wrappers. If any bare except exists in models/, it is likely a leftover from the copy. Replace with:
- `except sqlite3.IntegrityError as e:` for constraint violations
- `except sqlite3.OperationalError as e:` for schema/connection issues
- Or remove the try/except entirely if the model function should let the exception propagate to the caller

**Routes that genuinely need broad catch:**
Some route handlers wrap entire complex operations (multi-step server operations). If a broad catch is truly needed at the route level, use:
```python
except Exception as e:  # broad-catch-ok: multi-step operation, catch-all for user-facing error
    current_app.logger.error("Unexpected error: %s", traceback.format_exc())
    flash(f'An unexpected error occurred: {e}', 'danger')
    return redirect(url_for('servers.servers_list'))
```

Use this SPARINGLY — only where the alternative is an unhandled 500 with no user feedback.
